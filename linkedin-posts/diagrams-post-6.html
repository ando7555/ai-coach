Type Safety — String vs Enum
Catch invalid values at compile time, not in production

String fatigueLevel

"FRESH" "TIRED" "TRED" "" "maybe"

All compile. All valid. Bugs hide until runtime.

➔

FatigueLevel fatigueLevel

FRESH MODERATE TIRED EXHAUSTED

Only 4 valid values. Typos don't compile.

String injuryRisk

"LOW" "MEDIUM" "HIGH" "maybe high"

➔

InjuryRisk injuryRisk

LOW MEDIUM HIGH

String intensity

"LOW" "MEDIUM" "HIGH" "very high"

➔

TrainingIntensity intensity

LOW MEDIUM HIGH

String: "TRED" compiles silently
String: no IDE autocomplete
String: switch can miss cases
Enum: compile-time safety
Enum: IDE autocomplete
Enum: GraphQL validates input
Enum: switch exhaustiveness
Neo4j compatibility: SDN stores enums as their .name() string. FatigueLevel.FRESH → "FRESH" in DB. Existing data works without migration.

✓ After: 1 Helper

public Mono<String> generateTacticalAdvice(prompt) { return callClient(tacticalClient, prompt, "Tactical advice"); } public Mono<String> generateSeasonPlan(prompt) { return callClient(seasonPlanClient, prompt, "Season plan"); } public Mono<String> generateTrainingPlan(prompt) { return callClient(trainingPlanClient, prompt, "Training plan"); } private Mono<String> callClient( ChatClient client, String prompt, String context) { return Mono.fromCallable(() -> client.prompt().user(prompt) .call().content()) .subscribeOn(Schedulers.boundedElastic()) .doOnError(e -> log.error( "{} generation failed", context, e)) .onErrorResume(e -> Mono.empty()); }
