String Immutability — O(n²) Proof
Every += in a loop creates a brand-new object and copies all previous bytes

String += in Loop — O(n²×L)

1
copy L
copies L bytes, new object
2
copy 2×L
copies 2L bytes, new object
3
copy 3×L
copies 3L bytes, new object
4
copy 4×L
copies 4L bytes, new object
⋯
L + 2L + 3L + ⋯ + nL = L×(n×(n+1)/2)
result = "";
for (player : players) {
  result += formatPlayer(player); // NEW object every iteration!
}
Total copies: O(n²×L)

StringBuilder — O(n×L) Amortized

1
append L
writes into existing buffer
2
append L
same buffer, no copy
3
append L
same buffer, no copy
△
buffer doubles
O(n) amortized resize
⋯
L + L + L + ⋯ = n×L total writes
var sb = new StringBuilder();
for (player : players) {
  sb.append(formatPlayer(player));// in-place write
}
result = sb.toString();  // ONE final copy
Total writes: O(n×L) — no quadratic copies
Why strings are immutable: Java's String stores a final char[]. Every += calls new String(old + addition), allocating a fresh object on the heap and copying all previous bytes into it. With 25 players and prompts averaging ~200 chars (L=200): String concatenation copies ~62,500 bytes total. StringBuilder copies just ~5,000 bytes. That's a 12x reduction. StringBuilder uses a resizable char[] — it appends in-place and only copies when doubling capacity (amortized O(1) per append).
