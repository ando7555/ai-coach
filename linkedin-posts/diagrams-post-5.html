  • Text-based protocol
  • One request per connection
  • Keep-alive for reuse
  • Host header (virtual hosts)
  • Chunked transfer encoding
HTTP/2
2015
TCP + Binary
  • Binary framing layer
  • Multiplexing (many streams, one connection)
  • Header compression (HPACK)
  • Server push
  • Stream prioritization
HTTP/3
2022
QUIC over UDP
  • UDP instead of TCP
  • No head-of-line blocking
  • 0-RTT connection establishment
  • Built-in encryption (TLS 1.3)
  • Connection migration

HTTP, REST vs GraphQL, and the Spring Request Lifecycle

I'm building an AI-powered football coaching app called AI Coach. Every feature — tactical analysis, training plans, season planning — is exposed through a single GraphQL endpoint. No REST routes. No URL versioning. Just one POST to /graphql.

This article traces how the web works, why we chose GraphQL over REST, and what happens to every HTTP request as it flows through 11 layers of the Spring MVC stack.

HTTP — The Transport Layer

Every request follows the same path: DNS → TCP handshake → TLS (if HTTPS) → HTTP request.

  • HTTP/1.1 (1997) — text-based, one request per connection, keep-alive. AI Coach runs on this in development.
  • HTTP/2 (2015) — binary framing, multiplexing, HPACK header compression. Used behind reverse proxies.
  • HTTP/3 (2022) — QUIC over UDP, eliminates head-of-line blocking, 0-RTT connections. The future of web transport.
  • GraphQL only uses POST — the query is in the request body. REST uses GET, POST, PUT, DELETE, PATCH.
  • Always HTTP 200 — GraphQL handles errors in the response body, not status codes.

REST vs GraphQL — Why GraphQL Won

AI Coach is a textbook GraphQL fit: Neo4j graph database + deeply nested relationships + AI-generated content.

  • REST — 18+ endpoints for AI Coach, over-fetching, under-fetching, N+1 endpoint problem.
  • GraphQL — 1 endpoint, client specifies exact fields, typed schema, one round trip.
  • Schema — 13 queries + 9 mutations + 4 enums + 12 input/output types, all defined in schema.graphqls.

The Request Lifecycle — 11 Steps

Every generateMatchAnalysis mutation flows through:

  1. Tomcat receives TCP connection on port 8080
  2. JwtAuthenticationFilter extracts + validates Bearer token
  3. SecurityFilterChain: /graphql → permitAll()
  4. DispatcherServlet routes to GraphQL handler
  5. GraphQL engine parses query against typed schema
  6. @PreAuthorize("isAuthenticated()") blocks anonymous users
  7. @Valid triggers bean validation on input record
  8. CoachService: fetch from Neo4j → build prompt → call AI → save
  9. AiClient: reactive Mono on boundedElastic scheduler
  10. GraphQLExceptionHandler maps domain exceptions to ErrorType
  11. HTTP 200 JSON response: data or errors in body

Security — Layered Defense

  • JWT — HMAC-SHA256 signed tokens with username + role claims, 24h expiry.
  • CSRF disabled — stateless API with Bearer tokens, no cookie-based auth.
  • CORS configured — allows frontend origins (localhost:3000, 5173), preflight caching.
  • Method-level security — @PreAuthorize per resolver, not per URL (necessary for single-endpoint GraphQL).

Production Hardening — 4 Improvements

  • CORS — CorsConfigurationSource bean in SecurityConfig with allowed origins, methods, headers.
  • Actuator health — /actuator/health for Docker/Kubernetes liveness and readiness probes.
  • Profile-specific GraphiQL — application-prod.yaml disables GraphiQL and schema introspection.
  • Query depth limit — max-depth: 10 prevents DoS via deeply nested queries.

The bottom line: HTTP is the transport layer. GraphQL is the query language. Spring MVC is the framework. Security is layered. Every request flows through all four in sequence — understanding each layer separately makes the whole stack predictable and debuggable.
